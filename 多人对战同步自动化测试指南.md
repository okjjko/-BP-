# 多人对战同步自动化测试指南

## 概述

这个自动化测试将验证多人对战模式的实时同步功能。测试会自动启动三个浏览器上下文，模拟主办方和两个选手，并验证所有关键同步点。

## 测试场景

### 完整测试 (`完整的多人对战同步流程`)

测试流程：
1. ✅ 主办方创建房间
2. ✅ 选手1加入房间
3. ✅ 选手2加入房间
4. ✅ 主办方开始游戏
5. ✅ 验证身份分配同步
6. ✅ 选手1执行Ban操作
7. ✅ 验证所有客户端看到Ban结果
8. ✅ 验证控制台同步日志
9. ✅ 验证UI更新同步

### 快速验证测试

1. **快速验证：WebRTC连接建立**
   - 测试基础的P2P连接是否正常
   - 验证主办方能看到选手加入

2. **验证：身份分配消息**
   - 测试身份分配消息是否正确发送和接收
   - 验证控制台日志中的 `[gameStore] 收到身份分配消息`

## 运行测试

### 前置条件

```bash
# 1. 确保已安装依赖
npm install

# 2. 确保已安装 Playwright 浏览器
npx playwright install

# 3. 启动开发服务器（在一个终端窗口）
npm run dev
```

**重要提示**：
- 开发服务器默认运行在 `http://localhost:3000/`
- 如果运行 `npm run dev` 时提示 "Port 3000 is in use"，说明已经有服务器在运行
- 此时**不需要**再次启动服务器，直接运行测试即可

### 测试命令

在另一个终端窗口运行：

```bash
# 无头模式运行（后台运行，不显示浏览器）
npm run test:multiplayer

# 有头模式运行（显示浏览器窗口，可以看到自动化过程）
npm run test:multiplayer:headed

# 调试模式运行（逐步执行，方便调试）
npm run test:debug agents/tests/multiplayer-sync-test.spec.js
```

### 指定测试环境

```bash
# 使用自定义开发服务器URL
DEV_SERVER_URL=http://localhost:3000 npm run test:multiplayer
```

## 验证点

### ✅ 必须通过的验证

1. **WebRTC 连接成功**
   - 邀请码正确生成（6位字符）
   - 选手成功连接到主办方
   - 主办方能看到已连接的用户列表

2. **身份分配正确**
   - 选手1收到 `player1` 身份
   - 选手2收到 `player2` 身份
   - 控制台显示 `[gameStore] 收到身份分配消息`

3. **状态同步正常**
   - Ban操作后所有客户端状态一致
   - 控制台显示 `[gameStore] 收到状态更新 vX`
   - 控制台显示 `[gameStore] 已同步状态 vX`

4. **UI 更新同步**
   - BanArea 显示已禁用的植物
   - StageIndicator 显示正确的阶段
   - 同步状态指示器显示"已同步"（绿色）

### ⚠️ 可能失败的验证

如果测试失败，检查以下常见问题：

| 问题 | 可能原因 | 解决方法 |
|------|---------|---------|
| `选手应该已连接` 失败 | WebRTC连接失败 | 检查防火墙，确保 PeerJS 信令服务器可访问 |
| `选手应该收到身份分配消息` 失败 | `roomManager.js` 消息处理缺失 | 检查 `setupConnectionHandlers` 中的 `identityAssigned` 处理 |
| `应该有同步相关的日志` 失败 | `startStateSync()` 未被调用 | 检查 `createRoom()` 后是否调用了 `store.startStateSync()` |
| 状态不一致 | `handleStateUpdate()` 使用 `$patch()` | 改用对象替换模式确保深度响应式 |


## 常见问题

### Q1: 测试显示 "Test timeout of 30000ms exceeded"

**原因**: 测试超时，通常是因为 WebRTC 连接建立太慢。

**解决**:
1. 增加测试超时时间（在测试文件中）：
```javascript
test.setTimeout(120000); // 2分钟
```
2. 检查网络连接是否正常

### Q2: 主办方看不到选手连接

**原因**: PeerJS 连接失败。

**解决**:
1. 确认开发服务器正在运行
2. 检查浏览器控制台是否有 PeerJS 错误
3. 尝试重新安装 Playwright 浏览器：
```bash
npx playwright install --force
```

### Q3: 没有同步日志输出

**原因**: 事件监听器未正确设置。

**解决**:
1. 检查 `gameStore.js` 中的 `startStateSync()` 是否被调用
2. 检查 `roomManager.js` 是否正确处理 `identityAssigned` 消息
3. 参考修复日志确保所有修复已应用

### Q4: 测试通过但实际使用不同步

**原因**: 测试环境和真实环境差异。

**解决**:
1. 使用 `npm run test:multiplayer:headed` 观察测试过程
2. 对比测试截图和实际界面
3. 在真实浏览器中运行手动测试（参考 `多人对战同步测试指南.md`）

## 调试技巧

### 1. 使用有头模式观察测试

```bash
npm run test:multiplayer:headed
```

这样可以看到三个浏览器窗口的自动化操作过程。

### 2. 使用调试模式暂停测试

```bash
npm run test:debug agents/tests/multiplayer-sync-test.spec.js
```

测试会在每个步骤前暂停，方便检查状态。

### 3. 查看详细控制台日志

测试会自动收集并打印所有 `[gameStore]` 和 `[RoomManager]` 相关的日志。如果测试失败，首先查看控制台输出。

### 4. 检查截图

即使测试失败，截图也会被保存。检查 `agents/screenshots/multiplayer-sync/` 目录中的截图可以直观地看到失败点。

## 集成到 CI/CD

可以在 GitHub Actions 或其他 CI 系统中运行：

```yaml
- name: 启动开发服务器
  run: |
    npm run dev &
    sleep 10

- name: 运行多人对战同步测试
  run: npm run test:multiplayer

- name: 上传测试截图
  if: failure()
  uses: actions/upload-artifact@v3
  with:
    name: test-screenshots
    path: agents/screenshots/multiplayer-sync/
```
