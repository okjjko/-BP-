# 多人房间同步功能 - 新需求实现方案

## Context

在已实现的多人房间基础功能上，用户提出三个新需求以增强用户体验和数据一致性：

**新需求**：
1. **角色显示** - 在UI面板上显示用户当前的角色（主办方、选手1、选手2、观众）
2. **实时状态同步** - 确保所有操作立即同步到房间内的所有用户，无需刷新页面
3. **回合制权限控制** - 严格限制选手只能在各自的回合进行操作，防止越权操作

**用户建议方案**：
- 选手各自输入自己的ID来确认身份
- 根据选手ID判断当前回合，禁用非当前回合选手的操作

**当前系统状态**：
- ✅ 基础P2P连接已实现
- ✅ 房间创建/加入功能已实现
- ✅ 基础状态同步已实现
- ⚠️ 缺少角色UI显示
- ⚠️ 实时同步存在延迟
- ⚠️ 缺少严格的回合制权限控制

## 需求2：实时状态同步（立即同步到所有人）

### 目标
确保任何操作立即同步到房间内的所有用户，无需刷新页面。

### 当前问题
1. 状态同步通过主办方中转，存在延迟
2. 选手操作需要先发送到主办方，再由主办方广播
3. 存在网络延迟或主办方处理延迟

### 优化方案

#### 方案A：优化现有中心化同步（推荐用于简化实现）

**核心思路**：保持主办方为中心节点，但优化同步流程

**实现要点**：
1. 主办方收到状态更新后立即广播，不进行额外处理
2. 选手操作时，主办方更新状态后立即广播给所有人
3. 使用 `nextTick` 确保DOM更新后再同步

**修改文件**：`src/store/gameStore.js`

```javascript
syncState() {
  if (this.roomMode === 'local') return

  this.stateVersion++

  const gameState = { /* ... */ }

  if (this.roomMode === 'host') {
    // 主办方：立即广播
    nextTick(() => {
      roomManager.broadcastState(gameState, this.stateVersion)
    })
  } else {
    // 选手：发送到主办方
    roomManager.sendStateUpdate(gameState, this.stateVersion)
  }
}

// 主办方收到选手操作时，立即转发
handleStateUpdate(message) {
  const { senderId, senderRole, version, gameState } = message

  // 接受选手的更新
  this.stateVersion = version
  this.$patch(gameStateOnly)

  // 立即转发给其他所有连接的选手
  if (this.roomMode === 'host' && senderRole === 'player') {
    nextTick(() => {
      roomManager.broadcastState(gameState, this.stateVersion, senderId)
    })
  }
}
```

**修改文件**：`src/utils/roomManager.js`

```javascript
// 广播状态到所有连接（新增 excludePeerId 参数）
broadcastState(gameState, version, excludePeerId = null) {
  const message = {
    type: 'stateUpdate',
    senderId: this.peer.id,
    senderRole: this.role,
    timestamp: Date.now(),
    version,
    gameState
  }

  this.connections.forEach((conn, peerId) => {
    // 跳过发送者（避免回环）
    if (conn.open && peerId !== excludePeerId) {
      conn.send(message)
    }
  })
}
```

#### 方案B：P2P 全网状同步（高级方案）

**核心思路**：每个客户端都与其他客户端建立P2P连接

**优点**：
- 延迟最低，无需中转
- 更稳定，主办方断线不影响其他选手
- 实现真正的实时同步

**缺点**：
- 实现复杂度高
- 需要N*(N-1)/2条连接（N为房间人数）
- 需要更复杂的冲突解决机制

**建议**：先实现方案A，如果延迟问题严重再考虑方案B

## 需求3：回合制权限控制（严格权限）

### 目标
确保选手只能在各自的回合进行操作，防止越权操作。

### 当前问题
1. `hasBPPermission` 只检查是否为观众模式，不检查回合
2. 选手可以在任何时间点击操作（只要不是观众）
3. 没有身份绑定机制

### 解决方案：选手ID身份确认 + 严格权限控制

#### 步骤1：选手各自输入ID确认身份

**修改文件**：`src/components/RoomSetup.vue`

**选手加入房间流程**：
1. 选择"加入房间"
2. 输入邀请码
3. **新增**：输入自己的ID/昵称
4. 连接到主办方

**实现代码**：
```vue

  </div>

  <button @click="joinRoom">加入房间</button>

<script>
const playerName = ref('')

const joinRoom = async () => {
  if (!inputInviteCode.value || !playerName.value) {
    joinError.value = '请输入邀请码和你的ID'
    return
  }

  try {
    await roomManager.joinRoom(inputInviteCode.value, 'player', playerName.value)
    isConnected.value = true
    // 设置选手身份

    joinError.value = error.message
  }
}
```

#### 步骤2：在 gameStore 中添加身份绑定

**修改文件**：`src/store/gameStore.js`

```javascript
state: {
  // ... 现有状态

  // 新增：身份绑定
  myPlayerId: null,      // 'player1' | 'player2' （在多人模式下）
  myPlayerName: '',     // 选手ID/昵称
  myAssignedPlayer: null, // 分配的实际player标识（与BP回合绑定）
}

// 新增action
setMyIdentity(role, playerName) {
  this.myRole = role
  this.myPlayerName = playerName

  // 主办方设置
  if (role === 'host') {
    this.myPlayerId = 'host'
    return
  }

  // 选手/观众：等待主办方分配
  // 选手身份会在比赛开始后分配（player1 或 player2）
}

// 主办方分配选手身份
assignPlayerIdentity(clientPeerId, playerName, playerNumber) {
  // playerNumber: 'player1' | 'player2'

  // 找到对应的连接
  const conn = this.connections.find(c => c.peerId === clientPeerId)
  if (conn) {
    // 发送身份分配消息
    conn.send({
      type: 'identityAssigned',
      playerNumber: playerNumber,
      playerName
    })
  }
}
```

#### 步骤3：严格的回合制权限检查

**修改文件**：`src/store/gameStore.js`

```javascript
getters: {
  // 新增：判断是否为当前回合
  isMyTurn: (state) => {
    // 本地模式：总是可以操作
    if (state.roomMode === 'local') return true

    // 观众：不能操作
    if (state.isViewOnly) return false

    // 主办方：可以操作（用于测试或特殊情况）
    if (state.myRole === 'host') return true

    // 选手：检查是否为当前回合
    if (!state.currentRound?.currentPlayer) return false
    if (!state.myAssignedPlayer) return false

    return state.currentRound.currentPlayer === state.myAssignedPlayer
  },

  // 新增：获取我的操作权限描述
  myTurnDescription: (state) => {
    if (state.roomMode === 'local') return ''
    if (state.isViewOnly) return '观众模式'
    if (state.myRole === 'host') return '主办方'

    const currentPlayer = state.currentRound?.currentPlayer
    if (!currentPlayer) return '等待开始'

    if (state.isMyTurn) {
      const playerLabel = currentPlayer === 'player1' ? '选手1' : '选手2'
      return `当前回合：${playerLabel}`
    } else {
      const playerLabel = currentPlayer === 'player1' ? '选手1' : '选手2'
      return `对方回合：${playerLabel}`
    }
  }
}

actions: {
  // 修改 confirmSelection 添加权限检查
  confirmSelection() {
    // 新增：权限检查
    if (!this.isMyTurn) {
      alert(this.myTurnDescription || '现在不是你的回合！')
      return
    }

    if (!this.currentRound.selectedPlant) {
      alert('请先选择一个植物')
      return
    }

    // ... 原有逻辑
  }
}
```

## 数据一致性保证

### 状态同步时序

```
选手A操作 → 主办方接收 → 更新状态 → 广播到所有人
                          ↓
                    验证权限
                          ↓
                    状态更新
                          ↓
                立即同步到所有客户端
                          ↓
              所有客户端更新UI
```

### 冲突解决策略

1. **版本号机制**：每次状态更新递增 `stateVersion`
2. **时间戳验证**：较新的状态覆盖较旧的状态
3. **角色优先级**：主办方 > 选手 > 观众
4. **回合锁定**：只有当前回合的选手可以操作

### 防作弊机制

1. **客户端验证**：UI组件根据 `isMyTurn` 禁用按钮
2. **服务端验证**：主办方收到操作后验证权限
3. **操作日志**：记录所有操作，便于调试和审计
